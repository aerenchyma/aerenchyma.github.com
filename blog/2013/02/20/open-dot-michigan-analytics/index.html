Open.Michigan Analytics, the how (part 1)

Feb 20th, 2013

Okay, so I wanted analytics. That meant I needed to access and manipulate data, naturally. The phases laid out in the intial plan in that previous, linked, post were discussed during a meeting to kick off the project, the first day I came back to the office (I worked at a different job at Open.Michigan for nine months 2011 - 2012).

After the whats and the whys came the hows, and my job. What is the appropriate technical architecture for this task? (Spoiler: no Answers* here)

Facts:

  • OERbit/we run Drupal 6.x
  • Our servers shouldn’t take too much more load
  • We want to run fully open-source software and free software
  • MySQL is our db
  • Sustainability of and access to the analytics system matters
  • The base technical architecture for OERbit isn’t changing (at least, definitely not changing for this small piece)

Resources:

  • Python scripts that were written to gather Google Analytics and Youtube Analytics data (for the DataDive)
  • documentation of similar(?) analytics projects on Drupal
  • Drupal plugins galore?

Options/considerations:

  • Is there a Drupal plugin (or multiple) that suits our needs?
  • Can we customize a Drupal plugin to suit our needs? Or write a new one?
  • Should we write custom code? (Probably JavaScript)
  • Are there other options we’re not thinking about?
  • What useful (openly licensed) code is available? (Reuse, reuse, reuse)

And, in my mind: how can I use my skillset to work on this project most efficiently? At the start, I’d done very little programming in PHP, but was comfortable with Python and Ruby and reasonably comfortable with JavaScript. PHP is “like nothing else” and “exactly like all the other scripting languages, except with its own special things” (anonymity protected..). As I write, I’m still muddling around in the PHP world, hopefully to some success.

With an overarching goal of accessing the appropriate information and manipulating it, I had two immediate goals:

  1. Install a local instance of the OERbit software on my computer, for development
  2. Edit the extant Python scripts such that they got us the information we wanted, because they were existing resources

(1) proved to be much more difficult than any of us expected, but it was a learning process… which involved the office buying MAMP Pro and me installing the university-bought-license of the software on a loaner laptop, clean of my local db issues – at which point, with a great deal of help from the primary developers of OERbit, my life became much easier (the 2007 Macbook Pro is a pretty nice machine! If heavy /spoiltbyAir).

Documentation of this process (the way it should go) should appear on our wiki in future; documentation of the issues I have should appear internally (hopefully no one else will have to deal with exactly that. Be careful installing MySQL for funsies, all).

(2) was fun. More to come.

*Yes, the capital ‘A’ is on purpose
